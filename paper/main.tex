\begin{abstract}
Conditional Data Flow Diagrams (CDFDs) describe software behavior through data flows, processes, control conditions, and hierarchical process decomposition. Generating paths from a CDFD is therefore not equivalent to enumerating ordinary graph paths. A process may have several input and output ports, different ports may be mutually exclusive, inputs within one port may need to be available together, and independent branches may run concurrently. This paper presents a CDFD path generation tool that accepts a complete CDFD project file, converts it into a common graph model, checks structural validity, generates linear path projections and structured concurrent path expressions, and visualizes both the graph and the resulting path information. The paper defines the path model, explains JSON and SOFL `.cdfd' parsing, describes legality checks and system functions, details the port-aware algorithms for different CDFD cases, and evaluates the tool on representative CDFD inputs.
\end{abstract}

\begin{IEEEkeywords}
CDFD, SOFL, path generation, concurrent path, graph analysis, port semantics
\end{IEEEkeywords}

\section{Background and Significance}

Conditional Data Flow Diagrams (CDFDs) are used in SOFL-style specifications to describe how data items enter a system, are transformed by processes, are stored, and finally leave the system. Compared with a plain directed graph, a CDFD has richer semantics. Edges may represent data flow, active data flow, or control flow. Nodes may be processes, data stores, state nodes, condition nodes, or SOFL connector structures. A high-level process may also be decomposed into a lower-level CDFD.

The course task is to design a tool that can import an arbitrary CDFD following the agreed file format and automatically output its paths. The task requires the input file to contain all information needed by the tool, so the result should not depend on manual interpretation after import. It also requires a clear path definition because CDFD paths may involve concurrency and mutually exclusive process interfaces.

This topic is significant for two reasons. First, it turns an informal diagram-reading activity into a defined graph-analysis problem. A reader can ask whether a generated path is legal, why two branches are parallel, or why one input is insufficient to activate a process, and the tool can answer through explicit model rules. Second, it supports CDFD review and teaching. Instead of manually following arrows in a complex diagram, students can compare the imported CDFD, generated linear projections, structured concurrent path expressions, path relations, and consistency warnings.

\section{Method Overview}

This section defines the basic objects used by the tool and explains the overall method: input design, CDFD parsing, validity checking, path generation, and system output.

\subsection{CDFD Project Definition}

\textbf{Definition 1 (CDFD project).}
A CDFD project is a tuple
\begin{equation}
  P = (M, \mathcal{S}, G),
\end{equation}
where \(M\) is the module information, \(\mathcal{S}\) is the set of process specifications, and \(G\) is the CDFD graph to be analyzed.

\textbf{Definition 2 (CDFD graph).}
A CDFD graph is a tuple
\begin{equation}
  G = (V, E, \mathcal{D}, V_s, V_t, R),
\end{equation}
where \(V\) is the node set, \(E\) is the directed edge set, \(\mathcal{D}\) is the set of data items, \(V_s\) is the set of source nodes, \(V_t\) is the set of sink nodes, and \(R\) is the set of explicit structures such as parallel, fork, choice, join, and merge.

\textbf{Definition 3 (node).}
A node \(v\in V\) represents one CDFD element. Its type may be process, condition, data store, external entity, or connector. A process node may refer to a process specification in \(\mathcal{S}\). The graph \(G\) gives the topology among nodes, while \(\mathcal{S}\) gives the interface semantics of process nodes, such as inputs, outputs, ports, preconditions, and postconditions.

\textbf{Definition 4 (edge).}
An edge \(e \in E\) is represented as
\begin{equation}
  e=(src(e),dst(e),kind(e),data(e),guard(e)).
\end{equation}
Here \(src(e)\) is the source node, \(dst(e)\) is the target node, \(kind(e)\) is data-flow, active-flow, or control-flow, \(data(e)\subseteq\mathcal{D}\) is the data carried by the edge, and \(guard(e)\) is the condition under which the edge may be selected. If no guard is given, it is treated as true. Data-flow and active-flow edges are traversable by paths. Control-flow edges are not path segments; they contribute conditions to the affected process or path.

\subsection{Input and Output Design}

The tool accepts two input forms. The first is a graph-structured JSON file in the canonical `cdfd-json-v1' format. It records the module information, process specifications, graph nodes and edges, start and end nodes, ports, guards, and explicit structures.

The second input form is a `.cdfd' file generated by SOFLTool. This file is an XML document that stores a CDFD drawing through component and connection lists. The importer converts both JSON and SOFLTool `.cdfd' files into the same internal CDFD project model, so all later checks and path algorithms operate on one unified graph representation.

The tool output is the set of all paths generated from the input CDFD. Each path records the involved nodes, edges, carried data, selected conditions, and the corresponding source and sink.

\subsection{Basic Path Rules}

This subsection gives the basic rules used to decide whether a CDFD path is legal. The CDFD path itself is defined separately in the next subsection.

\textbf{Definition 5 (process port).}
For a process \(p\), an input port \(q^{in}\) is a group of input data items or input edges, and an output port \(q^{out}\) is a group of output data items or output edges. Inputs inside one input port use AND semantics by default. If the port mode is \texttt{any}, one available input is sufficient. Different input ports are activation alternatives unless the CDFD explicitly states that they must be synchronized. Different output ports are also alternatives by default, while multiple edges inside one selected output port may be produced together.

\textbf{Definition 6 (path state).}
During path generation, the current execution state is
\begin{equation}
  \sigma=(D,A,O,\Gamma),
\end{equation}
where \(D\subseteq\mathcal{D}\) is the set of available data items, \(A\subseteq V\) is the set of activated nodes, \(O\) maps activated nodes to their output results, and \(\Gamma\) is the set of collected path conditions.

\textbf{Definition 7 (port availability).}
Let \(I_q=In(p,q)\) be the set of traversable input edges assigned to input port \(q\) of process \(p\). We first define when a single input edge is available. In state \(\sigma=(D,A,O,\Gamma)\), an input edge \(e\) is available if its source node has already been reached and all data items carried by \(e\) are available:
\begin{equation}
  Avail(e,\sigma)\Leftrightarrow src(e)\in A \land data(e)\subseteq D.
\end{equation}
The port \(q\) is available according to its mode:
\begin{equation}
Avail(q,\sigma)\Leftrightarrow
\begin{cases}
\forall e\in I_q: Avail(e,\sigma), & mode(q)=all,\\
\exists e\in I_q: Avail(e,\sigma), & mode(q)=any.
\end{cases}
\end{equation}
Intuitively, an \texttt{all} port models synchronization: all required inputs must arrive before the process can use this port. An \texttt{any} port models alternative inputs: one available input edge is sufficient. If no explicit ports are defined, all input edges of \(p\) are treated as one \texttt{all} input port.

\textbf{Definition 8 (process activation).}
A process \(p\) is activatable in state \(\sigma\) when at least one of its input ports is available:
\begin{equation}
  Fire(p,\sigma)\Leftrightarrow
  \exists q^{in} \in \Pi^{in}_p: Avail(q^{in},\sigma).
\end{equation}
One activation selects an input port, selects an output port, and produces an output result:
\begin{equation}
  Activate(p,q^{in},q^{out},r).
\end{equation}
The selected output port determines which output edges may be produced together.

\textbf{Definition 9 (guarded edge).}
A traversable edge \(e\) may be followed in state \(\sigma=(D,A,O,\Gamma)\) only when its source node has been activated and its guard is satisfied:
\begin{equation}
  GuardOK(e,\sigma)\Leftrightarrow guard(e)(O(src(e)),D)=true.
\end{equation}
\begin{equation}
  Enabled(e,\sigma)\Leftrightarrow
  src(e)\in A \land GuardOK(e,\sigma).
\end{equation}
Thus a condition node or process output determines which branch is taken. If several guards from the same source are mutually exclusive, they define alternative possible paths; one concrete execution contains only the branch whose guard is true.

\subsection{CDFD Path Definition}

The central object of the method is the CDFD path. Unlike an ordinary graph path, a CDFD path is stateful: it must respect data availability, process ports, guarded branches, and synchronization.

\textbf{Definition 10 (CDFD path).}
A CDFD path describes one possible execution of a CDFD from source nodes to sink nodes. It keeps executed nodes and traversed edges as separate objects. Formally, a CDFD path is
\begin{equation}
  \pi=(V_\pi,E_\pi,\alpha_\pi,\prec_\pi,\Sigma_\pi),
\end{equation}
where:
\begin{itemize}
\item \(V_\pi\) is the finite set of node executions in the path;
\item \(E_\pi\subseteq E\) is the finite set of traversed data-flow or active-flow edges;
\item \(\alpha_\pi\) records the activation choice for each executed process, namely its selected input port, selected output port, and produced result;
\item \(\prec_\pi\) is the must-happen-before relation over node executions in \(V_\pi\). If \(u\prec_\pi v\), then \(u\) must execute before \(v\). Nodes with no ordering relation may execute in parallel;
\item \(\Sigma_\pi=(\sigma_0,\ldots,\sigma_n)\) is the state sequence produced by the execution, where each state has the form \(\sigma=(D,A,O,\Gamma)\).
\end{itemize}
\textbf{Definition 11 (path expression).}
For presentation, a path or a set of related paths is written as
\begin{equation}
  P ::= v \mid P_1 ; P_2 \mid Par(P_1,\ldots,P_k) \mid Xor(P_1,\ldots,P_k).
\end{equation}

\subsection{CDFD Parsing}

The JSON parser validates the file against the expected JSON schema and then creates `CDFDProject', `CDFDGraph', `Node', `Edge', `PortSpec', and `GraphStructure' objects. The SOFL parser reads XML components, resolves endpoints by component names and `shapeIndex', falls back to coordinate-based endpoint inference when a visible line touches a component, and preserves layout, port counts, connector indexes, and edge kinds as metadata.

Both importers normalize ports, guards, and explicit graph structures into the same CDFD project model.

After parsing, both JSON and SOFL inputs are represented as the same graph model:
\begin{equation}
  Input \rightarrow CDFDProject \rightarrow Path\ Algorithms.
\end{equation}

\subsection{Legality and Consistency Checks}

The tool performs two levels of validation. Schema validation is a hard check: if the JSON structure does not match the agreed format, the file is rejected. CDFD consistency checks are warnings because early CDFD models may be incomplete. Current checks include:

\begin{itemize}
\item process nodes without process specifications;
\item process specifications unused by any graph node;
\item graph data flows not declared in module variables;
\item process input/output declarations inconsistent with graph flows or ports;
\item disconnected data stores;
\item decomposition references pointing to missing graph layers;
\item cycles, reported separately from path generation.
\end{itemize}

\subsection{System Functions}

The implemented system provides these user-facing functions:

\begin{itemize}
\item import standard JSON and SOFL `.cdfd' files;
\item infer or accept explicit start and end nodes;
\item generate linear projections of valid CDFD paths;
\item generate structured path expressions with sequential, parallel, and exclusive branches;
\item analyze path relations such as parallel, exclusive, and joined-output;
\item expand multi-level process decompositions for linear path projections;
\item render SOFL-style SVG diagrams;
\item provide Web UI, Web API, and CLI output;
\item export text, JSON, CSV, and Markdown results.
\end{itemize}


\section{Key Technical Details}


\subsection{Overall Architecture}

The CDFD Path Generator follows a four-stage architecture: parsing, modeling, analysis, and output. A raw CDFD input, either in the canonical JSON format, in the custom `.cdfd' format, or in SOFL XML, is first parsed into the common project model \texttt{CDFDProject}. The analysis layer then runs linear path-projection enumeration, structured concurrent path search, multilevel expansion, path-relation inference, and consistency checking over the same model. The final output is a set of structured results, including \texttt{PathResult}, \texttt{ConcurrentPathResult}, and \texttt{FlowDecompositionResult}, which can then be exported or visualized.

The analysis pipeline contains two complementary path-generation routes. The first route performs depth-first search (DFS) to enumerate source-to-sink linear projections of valid CDFD paths. This route is compatible with traditional CDFD test scenarios and with multilevel process expansion. The second route uses a multi-token state machine to represent fork, join, and synchronized multi-input semantics as structured path expressions. This route addresses the limitation of linear projections, namely that a single sequence cannot express two independent branches running at the same time. Both routes share the same \texttt{CDFDGraph} model and the same node-activation predicate \texttt{can\_activate\_node}, so their semantics remain consistent.

\subsection{Data Model Design}

The system uses five model layers: the graph-element layer, the process layer, the project layer, the path-result layer, and the analysis-output layer. These layers are separated in responsibility but connected by explicit references, which makes them the common data foundation for all algorithms.

\subsubsection{Graph Element Layer}

The graph-element layer describes the topology of one CDFD graph. Its core classes are listed below.

\texttt{Node} stores a graph node. Its main fields are \texttt{id}, \texttt{type}, and \texttt{label}. The node type may represent a process, an external entity, or a data store.

\texttt{Edge} stores a directed connection. Its fields include \texttt{source}, \texttt{target}, \texttt{kind}, \texttt{data}, and \texttt{condition}; the condition is used as the edge guard in the path semantics. A control edge carries control information and is not treated as a traversable data-flow segment.

\texttt{PortSpec} describes a process port. Its fields include \texttt{data}, \texttt{edges}, and \texttt{mode}. The mode \texttt{all} represents AND semantics inside a port, while \texttt{any} represents OR semantics inside a port. This model is used to express SOFL processes with multiple interfaces, for example a login process that accepts either account-password input or token input.

\texttt{GraphStructure} stores explicit control structures, such as \texttt{parallel}, \texttt{choice}, and \texttt{join}. It prevents the later algorithms from relying only on topology guesses.

\texttt{CDFDGraph} is the container for one graph layer. It aggregates \texttt{nodes}, \texttt{edges}, \texttt{start}/\texttt{starts}, \texttt{ends}, \texttt{structures}, and \texttt{metadata}, such as output declarations and synchronized-start metadata.

Data-store connections are separated from side-port groups so that a data-store dependency is not accidentally treated as a mutually exclusive input.

\subsubsection{Process Layer}

\texttt{ProcessSpec} describes the interface of a process node. The fields \texttt{inputs} and \texttt{outputs} are compact declarations with AND semantics. A process may also produce several data items at once. The fields \texttt{input\_ports} and \texttt{output\_ports} represent explicit multi-port declarations. By default, different ports are alternatives, while data items within the same port must be supplied together unless the port states otherwise. The fields \texttt{pre} and \texttt{post} store textual preconditions and postconditions. The optional field \texttt{decom} can point to a child graph when multilevel expansion is enabled. \texttt{ModuleInfo} stores module-level information such as constants, types, and variables.

\subsubsection{Project Layer}

\texttt{CDFDProject} aggregates the module information, process specifications, and the CDFD graph analyzed by the core algorithms. In the implementation, decomposition graphs may be attached as an optional extension for multilevel expansion, but the basic method only needs one analysis graph.

\subsubsection{Path Result Layer}

\texttt{PathResult} records one source-to-sink linear projection of a valid CDFD path. It contains the node sequence, edge sequence, edge sources, edge targets, edge data, accumulated data, delivered outputs, explicit sink, preconditions, and edge conditions. In a multilevel result, an edge id is qualified by its graph name, such as \texttt{A1\_detail:e1}, so that every edge can still be traced to its original graph layer.

Structured concurrent path expressions are represented by a \texttt{ConcurrentPathNode} tree. A node with \texttt{kind=node} represents a graph node, \texttt{kind=sequential} represents sequential execution, and \texttt{kind=parallel} represents concurrent branches. \texttt{ConcurrentPathResult} keeps both the tree root and a flat summary of nodes, edges, data, outputs, and a readable notation such as \texttt{IN -> A -> [B || C] -> [OUT\_X4 || OUT\_X5]}.

\texttt{FlowDecompositionResult} stores all source-to-sink paths, detected cycles, and a source-to-sink flow-distribution matrix. The matrix records how many legal paths connect each source to each sink.

\subsubsection{Analysis Output Layer}

\texttt{PathRelation} describes relations among paths, such as \texttt{parallel}, \texttt{joined-output}, and \texttt{exclusive}. It also records the shared prefix and the reason for the relation. \texttt{FunctionalScenario} transforms one path or one concurrent group into a readable test scenario. \texttt{ConsistencyIssue} records warnings discovered by legality checks, such as missing process specifications, mismatched process inputs or outputs, and undeclared data flows.

\subsection{Node Activation Algorithm}

Before either the linear DFS or the concurrent token search can expand to a node, the algorithm must decide whether the node can be activated under the current data and activation state. This decision is centralized in \texttt{can\_activate\_node}. Its priority order is port-aware: explicit input ports are checked first, compact \texttt{inputs} declarations are checked second, and implicit port groups inferred from incoming edges are checked last.

\begin{widealgorithm}{CAN\_ACTIVATE\_NODE}
\Require graph $G$, node $v$, data set $D$, active set $A$, process map $P$
\Ensure \textbf{true} if $v$ can be activated; otherwise \textbf{false}
\State $process \gets P[v]$
\If{$process \ne \varnothing$ \textbf{and} $process.input\_ports \ne \emptyset$}
  \State $groups \gets \Call{GetInputPortGroups}{G,v,P}$
  \ForAll{$(port, group) \in \Call{Zip}{process.input\_ports, groups}$}
    \State $ready[port] \gets \Call{ExplicitPortReady}{port, group, D, A}$
  \EndFor
  \If{\Call{UsesOrInputPorts}{$process$}}
    \State \Return \Call{Any}{$ready$}
  \Else
    \State \Return \Call{All}{$ready$}
  \EndIf
\EndIf
\If{$process \ne \varnothing$ \textbf{and} $process.inputs \ne \emptyset$}
  \State \Return $process.inputs \subseteq D$
\EndIf
\State $(groups, ds\_edges) \gets \Call{GetInputPortGroups}{G,v,P}$
\If{$groups = \emptyset$ \textbf{and} $ds\_edges = \emptyset$}
  \State \Return \textbf{true}
\EndIf
\If{\Call{UsesOrInputPorts}{$G,v,P$}}
  \State \Return $\exists g \in groups:\Call{PortGroupReady}{g,D,A}$
\Else
  \State \Return $\forall g \in groups:\Call{PortGroupReady}{g,D,A}$
\EndIf
\end{widealgorithm}

The helper predicate \texttt{PORT\_GROUP\_READY} expresses AND semantics inside one port group: each required data item must already be available, and if the edge has a source node, that source must already be activated. The semantics between port groups are determined by \texttt{uses\_or\_input\_ports}. Mutually exclusive input alternatives use OR semantics, while synchronized join inputs use AND semantics.

\subsection{Linear Projection Enumeration Algorithm}

Linear projection enumeration uses DFS. Starting from each effective source, it extends through traversable flow edges and records a source-to-sink projection whenever it reaches a sink in \texttt{graph.ends}. The search is controlled by \texttt{PathFindingOptions}. In \texttt{simple} mode, repeated nodes are forbidden. In \texttt{max-depth} mode, repeated nodes are allowed only up to a depth bound. The global \texttt{max\_paths} bound prevents combinational explosion.

\begin{widealgorithm}{FIND\_PATHS}
\Require CDFD graph $G$, search options $O$, process map $P$
\Ensure all valid source-to-sink linear projections
\State $paths \gets \emptyset$
\Procedure{DFS}{$cur,nodePath,edgePath,dataPath,D,A,depth$}
  \If{$cur \in G.ends$}
    \State $outputs \gets \Call{NodeOutputs}{G,cur}$
    \State \Call{RecordPath}{$paths,nodePath,edgePath,dataPath,outputs,cur$}
    \If{$|paths| > O.max\_paths$}
      \State \textbf{raise} PathLimitExceeded
    \EndIf
    \State \Return
  \EndIf
  \If{$O.strategy=\textsc{max-depth}$ \textbf{and} $depth \ge O.max\_depth$}
    \State \Return
  \EndIf
  \ForAll{$edge \in \Call{OutgoingEdges}{G,cur}$}
    \If{$edge.kind=\textsc{control}$}
      \State \textbf{continue}
    \EndIf
    \If{$O.strategy=\textsc{simple}$ \textbf{and} $edge.target \in nodePath$}
      \State \textbf{continue}
    \EndIf
    \State $D' \gets D \cup edge.data$; $A' \gets A \cup \{cur\}$
    \If{\Call{RequiresGate}{$edge.target$}}
      \State $canEnter \gets \Call{CanActivateNode}{G,edge.target,D',A',P}$
      \If{\textbf{not} $canEnter$}
        \State \textbf{continue}
      \EndIf
    \EndIf
    \State \Call{CollectConditions}{$edge,conditions,preconditions$}
    \State $nextNodes \gets nodePath+[edge.target]$
    \State $nextEdges \gets edgePath+[edge.id]$
    \State $nextData \gets dataPath+edge.data$
    \State \Call{DFS}{$edge.target,nextNodes,nextEdges,nextData,D',A',depth+1$}
  \EndFor
\EndProcedure
\ForAll{$s \in \Call{EffectiveStarts}{G}$}
  \State \Call{DFS}{$s,[s],[],[],\emptyset,\emptyset,0$}
\EndFor
\State \Call{HandleStartsAndFilters}{$paths,G,P$}
\State $paths \gets \Call{DeduplicateAndSort}{paths}$
\State \Return $paths$
\end{widealgorithm}

Besides ordinary single-source DFS, the algorithm handles synchronized starts. If several start nodes jointly feed the same target through an AND input port, they are launched as one combined initial state. The SOFL importer also filters helper starts that only serve as auxiliary data channels and would otherwise produce redundant paths. Final results are deduplicated by nodes, edges, edge endpoints, edge data, accumulated data, and conditions.

\subsection{Concurrent Path Algorithm}

A linear path cannot express that two branches execute at the same time. Therefore the system introduces a multi-token state search. A token state records the current token locations, available data, activated nodes, traversed edges, and trace information for reconstructing the concurrent execution tree.

\begin{widealgorithm}{FIND\_CONCURRENT\_PATHS}
\Require CDFD graph $G$, search options $O$, process map $P$
\Ensure concurrent path expressions represented as execution trees
\State $frontier \gets \Call{InitialTokenStates}{G,P}$
\Statex \hspace{\algorithmicindent}\textit{single start or synchronized starts}
\State $results \gets \emptyset$; $visited \gets \emptyset$
\While{$frontier \ne \emptyset$}
  \State $state \gets \Call{Pop}{frontier}$
  \State $key \gets (state.tokens,state.available\_data)$
  \State $key \gets (key,state.activated\_nodes,state.edges)$
  \If{$key \in visited$}
    \State \textbf{continue}
  \EndIf
  \State $visited \gets visited \cup \{key\}$
  \If{\Call{IsConcurrentComplete}{$state,G$}}
    \State $results \gets results \cup \{\Call{StateToResult}{state,G,P}\}$
    \State \textbf{continue}
  \EndIf
  \State $nextStates \gets \emptyset$
  \ForAll{$token \in \Call{Sorted}{state.tokens}$}
    \If{$token \in G.ends$ \textbf{and} $token \in state.activated\_nodes$}
      \State \textbf{continue}
    \EndIf
    \State $outEdges \gets \Call{FlowOutgoingEdges}{G,token}$
    \If{\Call{ShouldForkParallel}{$token,outEdges$}}
      \State $forked \gets \Call{ApplyParallelFork}{state,token,outEdges}$
      \State $nextStates \gets nextStates \cup forked$
    \Else
      \ForAll{$edge \in outEdges$}
        \State $moved \gets \Call{ApplyEdgeTransition}{state,token,edge}$
        \State $nextStates \gets nextStates \cup moved$
      \EndFor
    \EndIf
  \EndFor
  \If{$nextStates=\emptyset$}
    \State $nextStates \gets \Call{ActivateReadyJoinNodes}{state,G,P}$
  \EndIf
  \State $frontier \gets frontier \cup nextStates$
\EndWhile
\State $keyFn \gets (sorted(edges),activated\_nodes)$
\State \Return \Call{DeduplicateBy}{$results,keyFn$}
\end{widealgorithm}

The initial token state depends on start synchronization. A normal graph creates one state for each start node. If the graph declares synchronized starts, or if a process requires inputs that come from several different start nodes, all starts are placed into one combined state. Parallel fork detection requires multiple outgoing flow edges, no explicit choice structure, compatible branch conditions, and no conflict between mutually exclusive output ports. Join activation is delayed until all required upstream branches have completed and the node-activation predicate returns true. After a concurrent result is constructed, the tree is normalized and formatted as a readable notation.

\subsection{Path Relation Inference Algorithm}

In addition to concurrent trees, the system infers relations among linear path projections. It recognizes parallel, joined-output, and exclusive relations. Explicit graph structures have priority. If the graph declares a parallel or choice structure, the relation is matched from that declaration. Otherwise, the system uses topology and condition compatibility.

\begin{widealgorithm}{BUILD\_PATH\_RELATIONS}
\Require linear path projection set $Paths$, CDFD graph $G$
\Ensure inferred path relations
\If{$G.structures \ne \emptyset$}
  \State $relations \gets \Call{MatchExplicitStructures}{Paths,G.structures}$
  \State \Return $relations$ \Comment{explicit structures have priority}
\EndIf
\State $relations \gets \emptyset$
\State $groups \gets \Call{GroupByOutputKey}{Paths}$
\ForAll{$g \in groups$ \textbf{with} $|g|\ge 2$}
  \If{\Call{SameEdgeConditions}{$g$} \textbf{and} \Call{PairwiseCompatible}{$g$}}
    \State $relations \gets relations \cup \{\Call{JoinedOutput}{g}\}$
  \EndIf
\EndFor
\State $H \gets$ empty compatibility graph over $Paths$
\ForAll{path pairs $(p_i,p_j)$ with $i<j$}
  \If{\Call{CanRunInParallel}{$p_i,p_j$}}
    \State \Call{AddEdge}{$H,p_i,p_j$}
  \EndIf
\EndFor
\State $cliques \gets \Call{BronKerbosch}{H}$
\ForAll{$C \in cliques$ \textbf{with} $|C|\ge 2$}
  \State $relations \gets relations \cup \{\Call{Parallel}{C}\}$
\EndFor
\State \Return $relations$
\end{widealgorithm}

For implicit parallel inference, two paths must have the same source, different outputs, compatible edge and control conditions, and disjoint node and data sets after their common prefix. The system then runs the Bron-Kerbosch maximal-clique algorithm on the compatibility graph so that a group of more than two mutually parallel paths is represented as one relation.

\subsection{Multilevel CDFD Expansion Algorithm}

When \texttt{ProcessSpec.decom} is not empty, the process node represents a child CDFD. The multilevel expansion algorithm first generates base paths on the parent graph and then replaces each decomposed process node with compatible child segments. A recursion stack prevents mutually recursive decompositions.

\begin{widealgorithm}{EXPAND\_GRAPH}
\Require project $Project$, graph name $name$, options $O$, expansion stack $S$
\Ensure linear projections after replacing decomposed processes with child CDFD paths
\If{$name \in S$}
  \State \textbf{raise} RecursionError \Comment{recursive decomposition}
\EndIf
\State $G \gets Project.graphs[name]$
\State $basePaths \gets \Call{FindPaths}{G,O,Project.processes}$
\State $expandedPaths \gets \emptyset$
\ForAll{$base \in basePaths$}
  \State $partials \gets \{\Call{EmptyPath}{base.preconditions}\}$
  \For{$i \gets 0$ \textbf{to} $|base.nodes|-1$}
    \State $node \gets base.nodes[i]$
    \State $segments \gets \Call{NodeSegments}{Project,node,S \cup \{name\}}$
    \If{$base.edges[i]$ exists}
      \State $parentData \gets base.edge\_data[i]$
      \State $segments \gets \Call{CompatibleSegments}{segments,parentData}$
    \EndIf
    \State $qualified \gets \Call{Qualify}{name,base.edges[i]}$
    \State $merged \gets \emptyset$
    \ForAll{$par \in partials, seg \in segments$}
      \State $merged \gets merged \cup \{\Call{Merge}{par,seg,qualified}\}$
    \EndFor
    \State $partials \gets merged$
  \EndFor
  \State $expandedPaths \gets expandedPaths \cup partials$
\EndFor
\State \Return $expandedPaths$
\end{widealgorithm}

The key step is compatibility filtering. If a parent edge carries output data such as \texttt{x7}, and a child graph has several internal branches with different outputs, only the child branch whose outputs intersect the parent edge data should be selected. If metadata is incomplete and no segment can be filtered confidently, the algorithm falls back to all candidate segments to avoid losing paths.

\subsection{Multi-Output and Flow Decomposition}

When a sink node can deliver several data items, or when a graph has several sinks, the system must determine the output data associated with each sink and summarize how paths are distributed among sinks.

\begin{widealgorithm}{COLLECT\_PATH\_OUTPUTS}
\Require CDFD graph $G$, sink node $t$, process map $P$
\Ensure output data items delivered at $t$
\If{$G.metadata.outputs[t]$ exists}
  \State $outputs \gets G.metadata.outputs[t]$
  \State \Return $outputs$
\EndIf
\ForAll{$edge \in \Call{IncomingEdges}{G,t}$}
  \State $process \gets \Call{SourceProcess}{edge,P}$
  \If{$process.output\_ports \ne \emptyset$}
    \If{\Call{MatchesOutputPort}{$edge,process.output\_ports$}}
      \State \Return $port.data \cup edge.data$
    \EndIf
  \EndIf
  \State $matched \gets edge.data \cap process.outputs$
  \If{$process.outputs \ne \emptyset$ \textbf{and} $matched \ne \emptyset$}
    \State \Return $edge.data \cap process.outputs$
  \EndIf
\EndFor
\State \Return $\bigcup_{edge \in \Call{IncomingEdges}{G,t}} edge.data$
\end{widealgorithm}

\texttt{decompose\_flow} wraps the linear projection set with sink-oriented summaries. It may sort paths by demand order, distance, or capacity metadata, detects cycles, and then builds a matrix from sources to sinks. It is not a max-flow algorithm; it summarizes legal enumerated paths rather than consuming shared edge capacity.

\subsection{Cycle Detection}

\begin{widealgorithm}{DETECT\_CYCLES}
\Require CDFD graph $G$
\Ensure all cycles in $G$ after ignoring control edges
\State $color[v] \gets \textsc{white}$ for each $v \in G.nodes$
\State $stack \gets []$; $cycles \gets \emptyset$
\Procedure{Visit}{$u$}
  \State $color[u] \gets \textsc{gray}$; \Call{Push}{$stack,u$}
  \ForAll{$edge \in \Call{FlowOutgoingEdges}{G,u}$}
    \State $v \gets edge.target$
    \If{$color[v]=\textsc{gray}$}
      \State $i \gets \Call{IndexOf}{stack,v}$
      \State $cycle \gets stack[i:] + [v]$
      \State $cycles \gets cycles \cup \{\Call{CanonicalRotation}{cycle}\}$
    \ElsIf{$color[v]=\textsc{white}$}
      \State \Call{Visit}{$v$}
    \EndIf
  \EndFor
  \State \Call{Pop}{$stack$}; $color[u] \gets \textsc{black}$
\EndProcedure
\ForAll{$v \in G.nodes$ \textbf{with} $color[v]=\textsc{white}$}
  \State \Call{Visit}{$v$}
\EndFor
\State \Return \Call{Deduplicate}{$cycles$}
\end{widealgorithm}

The algorithm uses a standard three-color DFS. Control edges are ignored because they contribute conditions rather than traversable data-flow segments. Cycles are normalized by canonical rotation so that the same cycle is not reported several times only because the DFS encountered it from different starting nodes.

\subsection{Functional Scenario Generation}

Functional scenario generation is the final transformation from path results to readable test-oriented descriptions. Concurrent path expressions that contain a parallel node become concurrent scenarios. Linear projections not covered by any concurrent scenario become single-path scenarios. This avoids describing the same execution flow twice.

\begin{widealgorithm}{BUILD\_FUNCTIONAL\_SCENARIOS}
\Require linear projections $Paths$, concurrent path expressions $CPaths$, path relations $Relations$
\Ensure readable functional scenarios
\State $scenarios \gets \emptyset$; $covered \gets \emptyset$
\ForAll{$cp \in CPaths$}
  \If{\Call{ContainsParallelNode}{$cp.root$}}
    \State $scenario \gets \Call{ConcurrentScenario}{cp,Relations}$
    \State $scenarios \gets scenarios \cup \{scenario\}$
    \State $covered \gets covered \cup \Call{CoveredPathIds}{Paths,cp}$
  \EndIf
\EndFor
\ForAll{$path \in Paths$}
  \If{$path.id \notin covered$}
    \State $scenario \gets \Call{SinglePathScenario}{path,Relations}$
    \State $scenarios \gets scenarios \cup \{scenario\}$
  \EndIf
\EndFor
\State \Return $scenarios$
\end{widealgorithm}

Coverage is checked by comparing edge sets after removing graph-name prefixes from multilevel edge ids. If a linear projection's edge set is a subset of the concurrent expression's edge set, it is treated as already covered by that concurrent scenario.

\subsection{Summary}

This section described the technical details of the CDFD Path Generator. At the model level, the graph-element, process, project, path-result, and analysis-output layers provide one shared representation for JSON, SOFL, and multilevel CDFDs. At the algorithm level, linear DFS and multi-token concurrent search form a dual path-generation framework. The unified activation predicate connects these two routes and keeps port semantics consistent. Additional mechanisms, including path-relation inference, multilevel branch filtering, output collection, and flow decomposition, turn raw path projections into structured business-flow descriptions.

\subsubsection{Linear Path}

A linear structure is the simplest CDFD case. It represents a data or control flow that proceeds in one direction without branching, looping, or path competition. In this case, the CDFD contains one source-to-sink path, and the path generator only needs to extract that single sequence.

\section{Uncertainties and Limitations}

Two points are not fully settled and should be stated explicitly.

First, SOFL `.cdfd' files store connector and coordinate information in tool-specific ways. The importer preserves connector indexes and infers missing endpoints, but more real SOFL files are needed to verify every port-mapping case.

Second, multi-level linear expansion is implemented, but fully unified concurrent expansion across nested decomposition graphs is not complete. A future version should replace a decomposed process with a concurrent subtree when the child graph contains parallel structure.

\section{Conclusion}

This paper presented a CDFD path generation method and tool that covers the required tasks: defining a complete input format, parsing CDFD data into a common graph model, checking legality and consistency, generating linear projections and structured path expressions, handling process ports and different algorithmic cases, and visualizing results. The main idea is to treat CDFD path generation as a semantic graph problem. Linear projections capture source-to-sink views of valid executions, while concurrent path expressions capture independent branches and synchronized joins. The port-aware rules prevent the common error of treating a multi-port process as if all of its interfaces were active at once. The implementation and case studies show that the approach can support JSON inputs, SOFL `.cdfd' imports, multilevel examples, port alternatives, joins, and cyclic graphs.
