\begin{abstract}
Conditional Data Flow Diagrams (CDFDs) are used to describe software behavior through data flows, processes, conditions, data stores, and hierarchical process descriptions. Generating all paths for a CDFD is therefore not the same as enumerating paths in an ordinary directed graph. A legal CDFD path must respect data availability, process inputs and outputs, mutually exclusive conditions, concurrent branches, and synchronization points. This paper presents a tool for generating all paths for a given CDFD. The paper first defines the CDFD path generation problem, then gives a path definition and generation rules for common CDFD structures, and finally describes the tool design and representative cases. The tool accepts CDFD input files, converts them into a unified graph model, checks basic consistency, generates linear and structured path results, and presents the results for review and later testing use.
\end{abstract}

\begin{IEEEkeywords}
CDFD, SOFL, path generation, conditional branch, concurrent path, graph analysis
\end{IEEEkeywords}

\section{Introduction}

A Conditional Data Flow Diagram (CDFD) is a graphical model for describing how data enters a software system, how it is transformed by processes, how conditions affect behavior, and how outputs are produced. In SOFL-style modeling, a CDFD is not only a drawing of connected components~\cite{liu1998sofl,liu2013formal}. It carries behavioral meaning: a process may require several input data items, an output may be produced only when a condition is satisfied, and several branches may be independent parts of the same execution.

The task studied in this paper is to design a tool that can generate all paths for a given CDFD. This task is important because paths are useful for understanding system behavior, checking whether a CDFD is complete, and deriving functional scenarios for testing. However, manual path tracing becomes unreliable when a CDFD contains conditions, multiple inputs, multiple outputs, data stores, and parallel branches.

The central challenge is that a CDFD path is not merely a sequence of graph nodes. In an ordinary directed graph, a path can usually be described as
\begin{equation}
v_0 \rightarrow v_1 \rightarrow \cdots \rightarrow v_k .
\end{equation}
For a CDFD, this sequence alone is not enough. We must also know which data items are available, which conditions hold, whether an input group is complete, and whether two branches are alternatives or concurrent parts of the same behavior.

This paper makes three contributions. First, it gives a clear definition of CDFD paths using common graph notation. Second, it summarizes path generation rules for the main CDFD structures, including linear flows, conditional branches, concurrent branches, joins, multiple inputs, and multiple outputs. Third, it describes a practical tool that imports CDFD files, builds a common internal model, generates path results, and visualizes both the CDFD and the generated paths.

\section{CDFD Path Generation Problem}

\subsection{Basic Elements of CDFD}

A CDFD contains several kinds of elements. A process represents a transformation from input data to output data. A data flow represents data moving from one element to another. A control flow represents a condition or control information that affects whether some behavior can occur. A data store represents persistent data that can be read or updated. A condition represents a decision that selects one or more branches. A source node provides input to the system, and a sink node receives output from the system. These ideas are related to the data-flow modeling tradition in structured analysis~\cite{demarco2011structured,yourdon1989modern}, while CDFD adds SOFL-specific conditions, process specifications, and decomposition rules~\cite{liu2004formal}.

\subsection{Differences from Ordinary Graph Paths}

An ordinary graph path is mainly a topological concept. It is valid when consecutive nodes are connected by directed edges. The meaning of the edge is usually not part of the path definition. A CDFD path is different because the graph topology and the diagram semantics must be considered together.

First, CDFD edges have different roles. A data-flow edge can be a path segment, while a control-flow edge usually contributes a condition instead of becoming a normal segment. Second, CDFD nodes are not always activated merely because they are reachable. A process can execute only when its required input data or selected input port is available. Third, outgoing branches have semantic relations. Branches guarded by different conditions are alternatives, while branches produced together by one process may be concurrent. Fourth, incoming branches may have to be synchronized before a later process can execute.

Thus, ordinary graph traversal only answers whether there is a connected route from one node to another~\cite{cormen2022introduction}. CDFD path generation must additionally decide whether the route is semantically executable and how its branch relations should be represented.

\subsection{Problem Statement}

Given a CDFD, the goal is to generate every legal source-to-sink behavior represented by the diagram. The input is a complete CDFD description containing nodes, flows, process interfaces, conditions, data stores, starts, and ends. The output is a set of paths. Each path records not only the nodes and edges it contains, but also the data and conditions that explain why the path is legal.

The problem can be summarized as follows:
\begin{equation}
\text{CDFD input} \rightarrow \text{legal path set}.
\end{equation}

A correct tool should be complete and sound for the CDFD structures supported in this paper. Completeness means that every legal path represented by the CDFD should be generated. Soundness means that every generated path should satisfy the CDFD semantics.

\section{Path Semantics and Generation Rules}

\subsection{Basic Notation}

In this paper, a CDFD is represented as a directed graph
\begin{equation}
G=(V,E),
\end{equation}
where \(V\) is the set of nodes and \(E\) is the set of directed edges. An edge is written as
\begin{equation}
e=(u,v),
\end{equation}
where \(u\in V\) is the source node and \(v\in V\) is the target node. The set of source nodes is denoted by \(S\), and the set of sink nodes is denoted by \(T\).

Each edge may carry a set of data items. The data carried by edge \(e\) is denoted by \(D(e)\). Some edges may also have a condition. The condition on edge \(e\) is denoted by \(C(e)\). If an edge has no condition, \(C(e)\) is treated as true.

For a process node \(p\), its required input data is denoted by \(In(p)\), and its produced output data is denoted by \(Out(p)\). If the process has ports, each port is treated as a group of related input or output data items. Data items inside one input port are usually required together, while different input ports usually represent alternative ways to activate the same process.

\subsection{Definition of a CDFD Path}

The key object of this paper is a CDFD path.

\textbf{Definition 1 (CDFD path).}
Given a CDFD \(G=(V,E)\), a CDFD path is written as
\begin{equation}
\pi=(N_\pi,E_\pi,D_\pi,C_\pi,R_\pi),
\end{equation}
where:
\begin{itemize}
\item \(N_\pi=(v_0,v_1,\ldots,v_k)\) is the node sequence;
\item \(E_\pi=(e_1,e_2,\ldots,e_k)\) is the edge sequence, where \(e_i=(v_{i-1},v_i)\);
\item \(D_\pi\) is the set of data items used or produced along the path;
\item \(C_\pi\) is the set of conditions collected along the path;
\item \(R_\pi\) records the relation among branches, such as sequence, choice, or concurrency.
\end{itemize}

A CDFD path must begin at a source node and end at a sink node:
\begin{equation}
v_0\in S,\quad v_k\in T .
\end{equation}
The path is legal only if every process on the path can be activated by the data available before it runs, and every selected conditional branch has a satisfiable condition.

This definition extends an ordinary graph path in two ways. First, it records data and conditions, because these decide whether the path is meaningful. Second, it records branch relations, because a CDFD may contain behavior that is better represented as a structured expression than as one flat sequence.

\subsection{Legal Path Rules}

A generated path is legal only when it satisfies the following rules.

\textbf{Rule 1: Traversable edge rule.}
Data-flow and active-flow edges may appear as path segments. Control-flow edges are not treated as ordinary path segments. Instead, they contribute conditions to \(C_\pi\) or affect whether a process can be activated.

\textbf{Rule 2: Data availability rule.}
When a path reaches a process \(p\), the data required by the selected input of \(p\) must already be available. If \(D\) is the data set currently available, then a simple process activation requires
\begin{equation}
In(p)\subseteq D .
\end{equation}
After \(p\) executes, its output data is added to the available data:
\begin{equation}
D := D\cup Out(p).
\end{equation}

\textbf{Rule 3: Port rule.}
If a process has input ports, each port is checked as one input group. For an input port \(q\), the port is available when all required data in that port has arrived:
\begin{equation}
In(q)\subseteq D .
\end{equation}
If several input ports are alternatives, one available port is enough to activate the process. If the CDFD marks them as synchronized inputs, all required ports must be available.

\textbf{Rule 4: Condition rule.}
If an edge \(e\) has a condition \(C(e)\), the path may include \(e\) only when that condition can hold. Conditions selected along the same path are collected in \(C_\pi\). If two outgoing branches have contradictory conditions, they represent different paths rather than one combined path.

\textbf{Rule 5: Branch relation rule.}
If several outgoing branches are controlled by mutually exclusive conditions, they are represented as choices. If several outgoing branches can be produced together and do not conflict, they are represented as concurrent branches. A structured path expression can therefore be written as
\begin{equation}
P ::= v \mid P_1;P_2 \mid Choice(P_1,\ldots,P_n) \mid Par(P_1,\ldots,P_n).
\end{equation}

\textbf{Rule 6: Join rule.}
If a process requires data from several upstream branches, the process can execute only after all required input data has arrived. In this case, the path generator must not treat each incoming branch as a complete independent behavior; it must synchronize them before continuing.

\subsection{Path Generation Procedure}

The core algorithm generates paths by exploring execution states, following the general idea of systematic graph search while adding CDFD-specific semantic checks~\cite{cormen2009}. A state records the active frontier, the activated nodes, the traversed edges, the available data, the collected conditions, and the structured expression built so far. A normal linear path is a special case in which the active frontier contains only one node. A concurrent path is represented by several active frontier nodes and by a \(Par\) expression.

\begin{paperalgorithm}{Generate All CDFD Paths}
\label{alg:generate-all}
\Require CDFD graph $G=(V,E)$, source set $S$, sink set $T$
\Ensure set of legal CDFD paths $P$
\State $P \gets \emptyset$; $Seen \gets \emptyset$
\State $Q \gets \Call{InitialStates}{S}$
\While{$Q$ is not empty}
  \State remove one state $(A,X,N,E_p,D,C,R)$ from $Q$
  \If{$(A,X,E_p,D,C) \in Seen$}
    \State \textbf{continue}
  \EndIf
  \State add $(A,X,E_p,D,C)$ to $Seen$
  \If{\Call{Complete}{$A,T$}}
    \State add \Call{NormalizePath}{$N,E_p,D,C,R$} to $P$
    \State \textbf{continue}
  \EndIf
  \State $Next \gets \emptyset$
  \ForAll{$v\in A$}
    \ForAll{branch case $B\in \Call{BuildOutgoingCases}{G,v,C}$}
      \State $base \gets (A-\{v\},X\cup\{v\},N,E_p,D,C,R)$
      \State $Cand \gets \{\Call{AppendBranch}{base,B}\}$
      \ForAll{edge $e=(v,u)\in B$}
        \State $New \gets \emptyset$
        \ForAll{$(A',X',N',E_p',D',C',R')\in Cand$}
          \State $E_p' \gets E_p'+(e)$
          \State $D' \gets D'\cup D(e)$; $C' \gets C'\cup C(e)$
          \If{\Call{CanActivate}{$u,D',X'$}}
            \State $Segs \gets \Call{Expand}{u,D',C'}$
            \ForAll{segment $H\in Segs$}
              \State $s \gets$ state after appending $H$
              \State add $s$ to $New$
            \EndFor
          \Else
            \State add $(A',X',N',E_p',D',C',R')$ to $New$
          \EndIf
        \EndFor
        \State $Cand \gets New$
      \EndFor
      \ForAll{$(A',X',N',E_p',D',C',R')\in Cand$}
        \State $J \gets \Call{ActivateReadyJoins}{G,A',X',N',D',R'}$
        \State $(A',X',N',R') \gets J$
        \State add $(A',X',N',E_p',D',C',R')$ to $Next$
      \EndFor
    \EndFor
  \EndFor
  \State add all states in $Next$ to $Q$
\EndWhile
\State \Return \Call{Deduplicate}{$P$}
\end{paperalgorithm}

In Algorithm~\ref{alg:generate-all}, \(A\) is the active frontier, \(X\) is the activated node set, \(N\) and \(E_p\) are the generated node and edge sequences, \(D\) is the available data set, \(C\) is the condition set, and \(R\) is the branch-relation expression. A state that cannot produce any successor is naturally discarded because no new state is added to \(Q\). Repeated states and repeated final paths are removed by \(Seen\) and \textsc{Deduplicate}.

\begin{paperalgorithm}{Build Outgoing Cases}
\label{alg:outgoing-cases}
\Require graph $G$, node $v$, current condition set $C$
\Ensure branch cases leaving $v$
\State $F \gets$ data-flow and active-flow edges leaving $v$
\State $K \gets$ control-flow information attached to $v$
\ForAll{$e\in F$}
  \State attach $C(e)$ and related conditions in $K$ to $e$
\EndFor
\State group $F$ by explicit structure and output port
\ForAll{group $g$}
  \If{$g$ is a choice or its guards are mutually exclusive}
    \State create one case for each edge in $g$
  \ElsIf{$g$ contains outputs produced together}
    \State create one concurrent case containing all edges in $g$
  \Else
    \State create one sequential case for each edge in $g$
  \EndIf
\EndFor
\State \Return all created cases
\end{paperalgorithm}

Algorithm~\ref{alg:outgoing-cases} explains how the tool distinguishes ordinary branches, conditional branches, and multiple outputs. Control-flow edges are not returned as path edges. They only add conditions to the branch cases. If several output edges belong to the same output port or explicit parallel structure, they become one concurrent case. If their guards are mutually exclusive, they become separate choice cases.

\begin{paperalgorithm}{Can Activate a Node}
\label{alg:can-activate}
\Require node $u$, available data $D$, activated node set $X$
\Ensure \textbf{true} if $u$ may execute
\If{$u$ is not a process}
  \State \Return \textbf{true}
\EndIf
\If{$u$ reads from or writes to a data store}
  \State require the data-store edge data to be available in $D$
\EndIf
\If{$u$ has explicit input ports}
  \ForAll{input port $q$ of $u$}
    \State $ready(q) \gets$ required data of $q$ is in $D$
    \State also require upstream sources of $q$ to be in $X$
  \EndFor
  \State \Return OR over alternative ports, AND over synchronized ports
\ElsIf{$In(u)$ is declared}
  \State \Return $In(u)\subseteq D$
\Else
  \State infer required input groups from non-control incoming edges
  \State \Return readiness of the inferred input groups
\EndIf
\end{paperalgorithm}

Algorithm~\ref{alg:can-activate} is the semantic gate of the path generator. It prevents a reachable process from being entered before its input data is available. It also handles AND semantics inside one input port and OR semantics between alternative input ports. When explicit process information is missing, the tool falls back to the incoming data-flow structure.

\begin{paperalgorithm}{Expand Decomposed Process}
\label{alg:expand-decomposed}
\Require process node $u$, available data $D$, condition set $C$
\Ensure path segment replacing $u$
\If{$u$ has no lower-level CDFD}
  \State \Return the single-node segment $(u)$
\EndIf
\State $G_u \gets$ lower-level CDFD of $u$
\State $P_u \gets \Call{GenerateAllCDFDPaths}{G_u}$
\State keep only child paths whose inputs are compatible with $D$
\State keep only child paths whose outputs match the parent flow
\State qualify child nodes and edges by the name of $u$
\State \Return the compatible child path segments
\end{paperalgorithm}

Algorithm~\ref{alg:expand-decomposed} handles hierarchical CDFDs. A decomposed process is not treated as a black box when detailed behavior is available. Instead, its internal paths replace the high-level node, while incompatible child paths are filtered out by input and output data.

\subsection{Generation Rules for CDFD Structures}

\subsubsection{Linear Flow}

A linear flow is the simplest case. Every node has one next step, and there is no branch selection or synchronization. The generated result is a single sequence:
\begin{equation}
v_0 \rightarrow v_1 \rightarrow \cdots \rightarrow v_k .
\end{equation}

\subsubsection{Conditional Branch}

A conditional branch appears when a condition selects one of several possible outgoing flows. If node \(p\) has two outgoing edges \(e_1\) and \(e_2\), and their conditions are \(C(e_1)\) and \(C(e_2)\), the tool creates separate path results:
\begin{equation}
\pi_1: p \rightarrow \cdots \quad \text{with } C(e_1),
\end{equation}
\begin{equation}
\pi_2: p \rightarrow \cdots \quad \text{with } C(e_2).
\end{equation}
The two paths should not be merged when the conditions are mutually exclusive.

\subsubsection{Concurrent Branch}

A concurrent branch appears when one process produces several outputs that can move through different branches independently. If the branches do not have conflicting conditions and are not marked as alternatives, the result should be represented as
\begin{equation}
Par(P_1,P_2,\ldots,P_n).
\end{equation}
This notation means that the branches are parts of the same behavior, not separate mutually exclusive paths.

\subsubsection{Join and Multiple Inputs}

A join appears when several branches provide data to the same later process. Suppose process \(p\) requires data \(d_1\) and \(d_2\), and these data items come from different branches. The process may execute only after
\begin{equation}
\{d_1,d_2\}\subseteq D .
\end{equation}
The generated path should therefore show that the upstream branches must be completed before \(p\) can run.

\subsubsection{Multiple Outputs}

A process may produce several output data items. These outputs may be sent to different sinks or later processes. If the outputs are produced together, the branches are concurrent. If they are produced under different conditions, they are alternatives. The tool distinguishes the two cases by checking output ports, edge conditions, and explicit structure information.

\subsubsection{Hierarchical Process}

Some CDFDs allow a process to be decomposed into a lower-level CDFD. In this case, the path generator first treats the high-level process as one node, and then replaces it with the paths generated from the lower-level diagram. The replacement is valid only when the output data of the lower-level path is compatible with the output data expected by the high-level edge.

\section{Tool Design}

\subsection{Overall Workflow}

The tool follows a five-step workflow:
\begin{enumerate}
\item import the CDFD file;
\item convert the input into a unified internal graph model;
\item check basic consistency of nodes, flows, data, and process interfaces;
\item generate legal path results according to the rules in Section III;
\item display or export the generated paths.
\end{enumerate}

The workflow is designed so that the path generation part does not depend on the original file format. Once the input has been converted into the internal model, JSON input and SOFL `.cdfd' input are handled in the same way.

\subsection{Input Design}

The tool supports two kinds of input. The first is a JSON format designed for this project. It records nodes, edges, data carried by edges, process specifications, source nodes, sink nodes, ports, and conditions. The second is a `.cdfd' file generated by SOFLTool. This file is parsed and converted into the same internal representation.

The input should contain enough information for automatic path generation. In particular, it should make clear which flows are data flows, which conditions guard branches, and which inputs or outputs belong to the same process port. If some information is missing, the tool uses conservative default rules and reports consistency warnings.

\subsection{Internal Model}

The internal model stores four kinds of information. First, it stores graph topology: nodes, directed edges, source nodes, and sink nodes. Second, it stores data information: data carried by edges, data required by processes, and data produced by processes. Third, it stores condition information: guards on outgoing branches and control information that affects process execution. Fourth, it stores structure information: whether branches are choices, concurrent branches, joins, or decomposed processes.

This separation is important because the same graph shape may have different CDFD meanings. For example, one node with two outgoing edges may be a conditional choice or a concurrent split. The tool should not decide this from topology alone when explicit semantic information is available.

\subsection{Path Generation Engine}

The path generation engine builds path results from source nodes to sink nodes. During generation, it maintains the current node, the edges already selected, the available data set \(D\), and the selected condition set \(C\). When the engine reaches a process, it checks whether the process input is available. When it reaches a conditional branch, it creates separate results for mutually exclusive choices. When it reaches concurrent branches, it records a structured relation instead of flattening the behavior into an arbitrary sequence.

The engine produces two complementary views. The first view is a linear projection, which is useful for reading source-to-sink paths and generating simple test scenarios. The second view is a structured path expression, which is useful for preserving concurrency and synchronization. For example, a generated expression may be written as
\begin{equation}
Start;Login;Par(CheckA,CheckB);Result .
\end{equation}

\subsection{Output Design}

Each generated path result contains the node sequence, edge sequence, data items, selected conditions, source node, sink node, and branch relation. The tool can present this information as text, JSON, CSV, Markdown, or a visual graph view. These outputs serve different purposes. Text and Markdown are useful for human reading. JSON and CSV are useful for later processing. The visual graph view helps users compare the original CDFD with the generated paths.

\subsection{Consistency Checking}

Before generating paths, the tool checks whether the CDFD description is sufficiently complete. Typical warnings include missing process specifications, data flows not declared in the module, process inputs not connected to any available data, outputs not connected to later nodes, and decomposition references that cannot be resolved. These checks do not replace human review, but they help locate modeling problems that would otherwise make path results confusing.

\section{Limitations and Future Work}

The current work focuses on finite forward CDFDs with explicit source-to-sink behavior. The quality of generated paths depends on the quality of the input model. If branch conditions, port information, or process outputs are missing, the tool can still generate paths using default rules, but the result may need manual review.

Another limitation is the interpretation of SOFLTool `.cdfd' files. Such files store drawing information and connector information in a tool-specific way. The importer can preserve the main graph structure, but more real project files are needed to improve the mapping between drawn connectors and semantic process ports.

Future work includes improving the visual editor, supporting richer scenario descriptions, providing stronger condition checking, and adding better explanations for why each path is generated. These improvements would make the tool more useful for requirements review and test-case derivation.

\section{Conclusion}

This paper presented a tool for generating all paths for a given CDFD. The main idea is that CDFD path generation should be treated as a semantic graph problem rather than a simple graph traversal problem. A legal CDFD path must respect data availability, process ports, conditions, branch relations, and synchronization points.

The paper defined CDFD paths using common notation, described generation rules for typical CDFD structures, and introduced a tool workflow for importing CDFD files, building a unified model, generating path results, and presenting them to users. By making path semantics explicit, the tool can help users understand complex CDFDs, check modeling consistency, and prepare functional scenarios for later testing work.
